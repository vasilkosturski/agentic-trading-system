# Storage Architecture: Agentic Trading System

## Overview

The agentic trading system uses **three distinct storage layers** to manage different types of data:

1. **PostgreSQL Database** - Primary persistent storage for all trading data
2. **Agent Memory (LibSQL)** - Per-agent SQLite databases for knowledge retention
3. **In-Memory Cache** - Volatile storage for market prices and frequently accessed data

---

## Layer 1: PostgreSQL Database (Primary Persistent Storage)

This is the main production database where all trading data lives. The schema is **auto-generated by Hibernate JPA** with three main schemas:

### Database Structure

```
PostgreSQL Database (agentic_trading)
├── agents schema
│   └── trading_agents table
│       ├── Agent metadata (name, description, risk_tolerance)
│       ├── Performance metrics (total_trades, total_pnl)
│       └── Activity tracking (last_activity, created_at, updated_at)
│
├── trading schema
│   ├── trading_accounts table
│   │   ├── Cash balance
│   │   ├── Agent reference (one-to-one)
│   │   └── Timestamps (created_at, updated_at)
│   │
│   ├── account_holdings table
│   │   ├── Current stock holdings (symbol, quantity, average_price)
│   │   └── Linked to trading_accounts
│   │
│   ├── account_transactions table
│   │   ├── Complete trade history (BUY/SELL)
│   │   ├── Trade metadata (symbol, quantity, price, timestamp)
│   │   ├── Rich decision context:
│   │   │   ├── rationale (brief reason)
│   │   │   ├── fullReasoning (comprehensive analysis)
│   │   │   ├── researchSources (JSON array of URLs researched)
│   │   │   └── agentContext (JSON snapshot of portfolio state)
│   │   └── Linked to agent runs for traceability
│   │
│   └── account_portfolio_snapshots table
│       ├── Daily portfolio snapshots (one per day)
│       ├── Holdings breakdown (total_value, holdings_value, cash_balance)
│       ├── P&L calculations (totalPnl, dailyPnl, totalReturnPercent)
│       └── Used for dashboard historical charts
│
└── analytics schema
    └── agent_runs table
        ├── Execution history (TRADING, REBALANCE)
        ├── Timestamp range (start_time, end_time)
        ├── Outcome tracking (TRADED, NO_TRADE, ERROR)
        ├── Rich decision data:
        │   ├── fullReasoning (agent's full thought process)
        │   ├── researchSources (JSON array of what was researched)
        │   ├── agentContext (JSON snapshot of portfolio state at run start)
        │   └── marketConditions (JSON market status at run start)
        ├── Performance metrics (tradeCount, errorMessage)
        └── Time-series data for monitoring agent behavior
```

### Entity Classes

Located in `backend/src/main/java/com/trading/entity/`:

- **`TradingAgent.java`** - Agent metadata, performance, and activity
- **`TradingAccount.java`** - Agent account with balance and account relationships
- **`AccountHolding.java`** - Current stock positions
- **`AccountTransaction.java`** - Complete transaction history with detailed context
- **`AccountPortfolioSnapshot.java`** - Daily portfolio snapshots for charting
- **`AgentRun.java`** - Agent execution records with full reasoning
- **`LogEntry.java`** - Logging and debugging information

### Automatic Schema Creation

Located in `backend/src/main/resources/application-postgresql.yml`:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://${DB_HOST:localhost}:${DB_PORT:5432}/${DB_NAME:agentic_trading}
    username: ${DB_USERNAME:trading_user}
    password: ${DB_PASSWORD:trading_password}
    
  jpa:
    database-platform: org.hibernate.dialect.PostgreSQLDialect
    hibernate:
      ddl-auto: update  # Auto-creates/updates schema on startup
    properties:
      hibernate:
        hbm2ddl:
          create_namespaces: true  # Automatically creates schemas
```

**Key Point:** You **never need to run SQL scripts**—Hibernate handles all schema creation and migrations automatically on startup.

## Layer 2: Agent Memory (LibSQL - Per-Agent SQLite Databases)

Each agent has its own **isolated memory database** for storing knowledge and context.

### Storage Location

```
/agentic-trading-system/agents/memory/
├── warren.db      # Warren's memory (SQLite)
├── george.db      # George's memory (SQLite)
├── ray.db         # Ray's memory (SQLite)
└── cathie.db      # Cathie's memory (SQLite)
```

### How It Works

- Uses **MCP Memory Server** (`mcp-memory-libsql`) - a third-party MCP providing a knowledge graph interface
- Each agent stores and recalls information about:
  - Companies they've analyzed
  - Market conditions observed
  - Websites of interest
  - Past research findings
  
- **No vector embeddings**—it's a simple key-value store (not FAISS, Pinecone, or ChromaDB)
- Designed for within-session context retention

### Configuration

From `agents/mcp_params.py`:

```python
def researcher_mcp_server_params(name: str):
    """
    MCP servers for researcher tool - EXTERNAL services only
    
    Args:
        name: Agent name for unique memory database
    
    Returns:
        List of MCP server parameter dicts
    """
    return [
        # Fetch - Standard web content retrieval
        {"command": "uvx", "args": ["mcp-server-fetch"]},

        # Brave Search - Third-party web search
        {
            "command": "npx",
            "args": ["-y", "@modelcontextprotocol/server-brave-search"],
            "env": brave_env,
        },

        # Memory - Third-party context storage
        {
            "command": "npx",
            "args": ["-y", "mcp-memory-libsql"],
            "env": {"LIBSQL_URL": f"file:./memory/{name}.db"},
        },
    ]
```

From `agents/simple_trader.py`:


### Important Notes

- **Semi-persistent**: Memory persists across agent runs but is not critical for operation
- **Per-agent isolation**: Each agent has separate memory, no shared knowledge graph
- **Not a vector store**: Simple key-value storage, not semantic search
- **Ephemeral safety**: Losing memory doesn't affect trading operation (all trades logged in PostgreSQL)

## Layer 3: In-Memory Cache (MarketService - Volatile)

Market prices are cached in Java memory to avoid excessive API calls to Polygon.io.

### Cache Implementation

From `backend/src/main/java/com/trading/service/MarketService.java`:

```java
// Price cache in memory (ConcurrentHashMap)
private final Map<String, CachedPrice> priceCache = new ConcurrentHashMap<>();

// TTL configurable (default: 60 minutes)
@Value("${market.cache.ttl-minutes:60}")
private int cacheTtlMinutes;
```

### Why This Works

- **Data Freshness**: Polygon.io free tier provides **end-of-day data only** (prices don't change intraday)
- **Cache Alignment**: 60-minute cache matches end-of-day data update frequency
- **Scale**: System has ~20-40 unique symbols across 4 agents
- **API Usage**:
  - Initial dashboard load: 20-40 API calls (cold cache)
  - Next 59 minutes: 0 API calls (served from cache)
  - Average: <1 API call/minute (well under Polygon's 5/min free tier limit)

### Cache Configuration

In `application-postgresql.yml`:

```yaml
market:
  cache:
    ttl-minutes: 60  # Default: 60 minutes
```

### CachedPrice Structure

```java
class CachedPrice {
    private double price;
    private Instant timestamp;
    
    public boolean isExpired(int ttlMinutes) {
        return Instant.now().isAfter(
            timestamp.plus(ttlMinutes, ChronoUnit.MINUTES)
        );
    }
}
```

## Important Notes

### No Vector Store

The system **does NOT use**:
- ❌ FAISS
- ❌ Pinecone
- ❌ ChromaDB
- ❌ Milvus
- ❌ Weaviate
- ❌ Vector embeddings

This design choice prioritizes:
- **Simplicity**: Fewer dependencies, easier to deploy
- **Performance**: No embedding model latency
- **Cost**: No embedding API costs
- **Clarity**: Direct trade reasoning logged as text

### When to Add Vector Storage

Consider adding a vector store if you need:
- Semantic search across past trades
- RAG (Retrieval-Augmented Generation) for agent decisions
- Similarity-based stock recommendation
- Pattern detection across historical trades

### Timestamp Handling

All timestamps in PostgreSQL are stored as `Instant` (UTC):
- Java entity type: `java.time.Instant`
- Database type: `TIMESTAMP`
- Frontend: Displays in user's local timezone automatically

## Performance Considerations

### Database Performance

- **Connection Pool**: 20 max, 5 min (HikariCP optimized)
- **Batch Operations**: Enabled for inserts/updates
- **Lazy Loading**: Relationships use `FetchType.LAZY` to avoid N+1 queries
- **Indexes**: Recommended on `account_id`, `agent_id`, `timestamp` columns

### Cache Performance

- **Price Cache**: ConcurrentHashMap for thread-safe access
- **TTL**: 60 minutes (configurable)
- **Expiration**: Lazy evaluation on cache access
- **Memory**: Minimal footprint (~20-40 entries at typical scale)

### API Performance

- **Response Time**: <100ms typical (cached prices)
- **Rate Limiting**: Polygon.io 5 calls/min (easily achieved)
- **Dashboard Refresh**: 15 seconds (all data cached, no DB hits)

## Backup and Recovery

### PostgreSQL Backups

```bash
# Full backup
pg_dump -h localhost -U trading_user agentic_trading > backup.sql

# Restore
psql -h localhost -U trading_user agentic_trading < backup.sql

# Using pg_basebackup (Kubernetes)
kubectl exec -it statefulset/postgres -n agentic-trading -- \
  pg_basebackup -D /backup -F tar -z
```

### Memory Database Backups

```bash
# Simply copy the memory database files
cp -r agents/memory/ agents/memory.backup/
```

### Restoration Strategy

1. **Critical**: PostgreSQL (all trading history)
2. **Important**: Agent runs and decisions
3. **Optional**: Agent memory databases (can be rebuilt)

## Summary

The agentic trading system uses a **layered storage architecture**:

1. **PostgreSQL**: Permanent trade data, decision logs, performance metrics
2. **Agent Memory**: Per-agent context and knowledge retention
3. **In-Memory Cache**: High-performance price lookups

This design balances **persistence, performance, and simplicity** without over-complicating with unnecessary vector embeddings or complex caching strategies.
